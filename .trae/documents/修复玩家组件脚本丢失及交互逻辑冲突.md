### 1. 问题分析 (Analysis)

经过代码和场景文件的深入审查，我确认了玩家无法选中和移动的根本原因，以及 TileInfo 面板显示延迟的潜在原因。

#### A. 玩家选中与移动失效的原因
**根本原因**: `Scenes/Entities/Player.tscn` 场景文件中的组件节点（`MovementComponent`, `HealthComponent`, `InteractableComponent`）**只是普通的 Node 节点，没有挂载对应的脚本**。

*   **Player.gd**: 使用了强类型变量 `@onready var movement_component: MovementComponent = $MovementComponent`。
*   **运行时行为**: 当 `Player.gd` 初始化时，它尝试将 `$MovementComponent` 转换为 `MovementComponent` 类型。由于该节点没有挂载脚本，类型转换失败（得到 `null`），或者如果 Godot 没有报错，后续调用 `movement_component.move_to()` 时也会因为 `null` 引用而失效或崩溃。
*   **InteractableComponent**: 同样没有挂载脚本。这意味着 `InteractionManager` 即使射线击中了它，尝试调用其方法（如 `try_interact`）也会失败，且其 `collision_layer` 初始设置（在 `_ready` 中设置）也不会执行，导致物理层级配置失效。

#### B. TileInfo 面板显示延迟的原因
**原因分析**: TileInfo 面板的更新依赖于鼠标移动事件。
*   **事件链**: `InputManager` -> `_handle_mouse_motion` -> `InteractionManager` (通常不直接处理 TileInfo) -> `WorldManager` 或 UI 系统。
*   **延迟来源**: `InputManager.gd` 中并没有发现直接导致延迟的逻辑。但值得注意的是，**鼠标拖拽阈值 (`drag_threshold`)** 确实会影响**点击事件**的触发时机（必须等松开鼠标才触发），但理论上不应影响**鼠标悬停 (Hover)** 信息的更新。
*   **可能的因素**:
    1.  **信号频率**: 如果 UI 更新连接的是 `camera_pan` 或其他低频信号。
    2.  **物理检测频率**: 如果 TileInfo 依赖于物理射线检测 (`_physics_process`)，那么它受限于物理帧率（默认 60fps），可能会有轻微滞后。
    3.  **UI 阻挡**: `InputManager` 有 `is_mouse_over_ui()` 检查。如果 TileInfo 面板本身被判定为“鼠标在其上的 UI”，可能会导致某些逻辑被阻断。
*   **当前结论**: 优先修复脚本挂载问题，因为这是严重的逻辑阻断错误。TileInfo 问题可能在组件修复后会有改善（如果它依赖于某些组件的状态），或者我们可以之后单独优化。

### 2. 解决方案 (Solution)

重点修复 `Player.tscn` 的脚本挂载问题。

#### **步骤 1: 挂载脚本**
修改 `Scenes/Entities/Player.tscn` 文件，将对应的 GDScript 脚本资源附加到节点上。

1.  **MovementComponent**:
    *   附加脚本: `res://Scripts/components/entity/MovementComponent.gd`
2.  **HealthComponent**:
    *   附加脚本: `res://Scripts/components/entity/HealthComponent.gd`
3.  **InteractableComponent**:
    *   附加脚本: `res://Scripts/components/entity/InteractableComponent.gd`
    *   **注意**: `InteractableComponent.gd` 在 `_ready()` 中会将 `collision_layer` 强制设置为 2。这可能会覆盖我们在上一步中手动设置的 Layer 8 (Entities)。**需要同步修改该脚本，使其兼容 Layer 8，或者在场景中覆盖它。** 考虑到之前决定玩家属于 Layer 4 (Value 8)，我们应该修改脚本中的默认值，或者在 `_ready` 中逻辑改为“如果不为 0 则保留，否则设默认值”。

#### **步骤 2: 修正 InteractableComponent.gd**
*   **问题**: `InteractableComponent.gd` 的 `_ready` 函数中有一行 `collision_layer = 2`。这会强制将层级重置为 Interactables (Layer 2)，导致之前设置的 Entities (Layer 4) 失效。
*   **修改**: 移除或修改这行强制赋值代码，允许在编辑器中配置层级。
